# horus-os Phase 30 Plan 01

## What the smoke script currently does (v0.1 + v0.2)

The Phase 20 script exposes `main() -> int`. Eighteen checks run in
one `try:` block sharing one tempdir. See the Phase 20 SUMMARY for the
list. All checks are kept verbatim.

## What v0.3 needs added

Inserted after the existing v0.2 public-surface import smoke (so the
final summary line stays accurate), one new block per area:

- **Lifecycle + registry imports**: extend the existing
  PUBLIC_SURFACE_IMPORT_SNIPPET so it also imports `LifecycleAdapter`,
  `AdapterRegistry`, `AdapterEntry` from `horus_os`, AND
  `DiscordAdapter`, `SlackAdapter`, `EmailAdapter`, `CalendarAdapter`,
  `LifecycleAdapter` from `horus_os.adapters`. Same subprocess; one
  exit code covers all of them. Assert `discover_adapters()` returns
  at least one entry (webhook is always there; the four others are
  present when .[all] is installed, which the CI job does).

- **Per-module direct imports**: a separate subprocess that imports
  each of the four adapter modules directly via their fully-qualified
  module path. This catches the case where one adapter module has a
  top-level SDK import that breaks when the SDK is missing. The
  modules use the lazy-import pattern, so this should succeed even
  without the optional extras installed.

- **GET /api/adapters via TestClient**: a separate subprocess builds
  `create_app(tmp_data_dir)`, wraps it in `TestClient`, hits
  `GET /api/adapters`, parses the JSON body, asserts the response is
  200, asserts the body has an `adapters` key whose value is a list,
  and asserts each entry carries the six required keys (name, status,
  last_activity_at, error_count, error_message, supports_toggle).
  Runs in a subprocess so the smoke driver process stays hermetic
  (FastAPI is an optional dep).

- **POST /api/adapters/{name}/disable 404 path**: same subprocess as
  the TestClient block above. Posts against a deliberately-bogus
  adapter name and asserts a 404. This confirms the toggle route is
  mounted end-to-end without requiring a real lifecycle hook (which
  would need real tokens).

## Stubbing patterns to reuse

### Pattern 1: subprocess via sys.executable

Same `run_python(snippet)` helper from Phase 20. No change needed.

### Pattern 2: TestClient in a subprocess

The subprocess body is a self-contained Python snippet that:

1. `from horus_os.server import create_app`
2. `from fastapi.testclient import TestClient`
3. `import tempfile; from pathlib import Path`
4. `d = Path(tempfile.mkdtemp(prefix='horus-os-smoke-app-'))`
5. Calls `horus-os init --data-dir d` via subprocess so the data dir
   has a database (TestClient needs a real `horus.sqlite`)... actually
   simpler: the smoke script already inited `tmp_dir`. Pass that dir
   in via an env var (or write the snippet to a tempfile and pass the
   path as argv). Argv is cleanest.
6. Uses `with TestClient(create_app(data_dir)) as client:` so the
   lifespan hooks run cleanly. Adapters that fail their `start` hook
   (Discord/Slack/Email/Calendar without tokens) bump the registry
   error counter but never abort the test (start failures are caught
   per Phase 22).
7. Asserts on `response.json()` shape, prints a single marker line.

### Pattern 3: Lazy-import-friendly per-module smoke

A single subprocess that runs `import horus_os.adapters.discord_adapter`
and the three siblings, then prints a marker. Catches the regression
where a top-level `import discord` slips in.
