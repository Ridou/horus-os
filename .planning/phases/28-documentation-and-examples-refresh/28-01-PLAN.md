## Stubbing pattern for offline examples

Same pattern the adapter test suite uses:

- Discord: `sys.modules["discord"] = types.ModuleType("discord")`
  with `Intents` and `Client` fakes attached. `run_agent`
  monkeypatched at the adapter module level.
- Slack: `sys.modules["slack_sdk"] = ...` with a fake `WebClient`
  that records `chat_postMessage`. HMAC signature is computed in
  the example using the same hmac.compare_digest path the adapter
  re-runs at request time.
- Email: `monkeypatch.setattr(email_adapter_module.imaplib,
  "IMAP4_SSL", _FakeIMAP)` plus the SMTP equivalent. Examples use
  the same approach (direct setattr, not monkeypatch).
- Calendar: fake `google`, `google.auth`, `google.auth.transport`,
  `google.auth.transport.requests`, `google.oauth2`,
  `google.oauth2.credentials`, `googleapiclient`,
  `googleapiclient.discovery` modules in sys.modules.

For each script the docstring documents the env vars an operator
sets for a live run.

## Migration guide shape

Mirror `docs/MIGRATION-v0.1-to-v0.2.md`:

- TL;DR: v0.3 is purely additive. Every v0.2 surface continues to
  work byte-identical. No schema migration.
- What is new: Protocol additions (LifecycleAdapter, AdapterRegistry,
  AdapterContext.tool_registry); four shipped adapters; new optional
  extras (discord, slack, calendar); new env vars (full list); new
  server routes; Dashboard Adapters tab.
- Schema migration: none.
- User-visible behavior changes: none for existing v0.2 code paths.
  New surfaces are opt-in.
- Deprecations: none.
- Downgrade: trivial (no schema changes); just `pip install horus-os==0.2.0`.
- Upgrade code samples: implementing a LifecycleAdapter; reading
  /api/adapters; configuring each shipped adapter.
