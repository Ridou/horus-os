<v0_4_baseline_json_canonical>
The exact JSON shape committed by this phase. Schema is additive over v0.3 (the four v0.3 fields stay; two new fields land). Implementing executor MUST produce this surface; do not invent fields, do not omit fields.

Top-level:
```
{
  "samples": [ ... ]
}
```

Each entry in `samples`:
```
{
  "os": "<linux | darwin | win32>",
  "python": "<3.11 | 3.12 | 3.13>",
  "horus_os_version": "0.4.0",
  "wall_clock_ms": <float, median of N=20 cold-start runs of create_app(), or null for placeholder rows>,
  "agent_loop_3_iter_ms": <float, median of N=20 3-iteration agent loops against the stub provider, or null for placeholder rows; preserved for v0.3 trend parity>,
  "cold_import_ms": <float, median of N=20 cold imports of horus_os.adapters in subprocess, or null for placeholder rows>,
  "entry_points_discovery_ms": <float, median of N=20 discover_adapters() calls in-process with sys.modules clearing between runs, or null for placeholder rows>,
  "n_samples": 20,
  "captured_at": "<ISO8601 with Z suffix at capture time, or 'placeholder' for placeholder rows>"
}
```

Field semantics:
- `wall_clock_ms`: total wall clock from `python -c "from horus_os.server.api import create_app; create_app()"` start to interpreter exit, measured by the parent process via subprocess.run + perf_counter_ns. This is the headline BASELINE-02 number (ROADMAP Phase 40 Success Criterion #1).
- `agent_loop_3_iter_ms`: same shape as v0.3 `median_3_iteration_loop_ms`. Carry over the v0.3 stub-Conversation + run_agent_loop pattern verbatim so the v0.3→v0.4 trend is computable in Phase 42's benchmark report. Renamed to `agent_loop_3_iter_ms` (the v0.3 name was `median_3_iteration_loop_ms`) to make the field semantics explicit on schema-only inspection.
- `cold_import_ms`: subprocess wall clock from `python -c "import horus_os.adapters"` start to exit. Measures the cold-import cost of the adapters package, which is the package Phase 42's discovery system extends.
- `entry_points_discovery_ms`: in-process wall clock for one call to `discover_adapters()`. The script clears `sys.modules` entries matching `horus_os.adapters*` between samples and re-imports to keep each sample comparable. With zero installed adapter packages on the maintainer's machine this returns `[]` immediately and the number captures the importlib.metadata + sort + iteration overhead floor.

Placeholder rows (rows for (os, python) combos to be backfilled later by CI runners):
- `os` + `python` + `horus_os_version` + `n_samples` are required and non-null
- All four measurement fields (`wall_clock_ms`, `agent_loop_3_iter_ms`, `cold_import_ms`, `entry_points_discovery_ms`) are JSON null
- `captured_at` is the literal string `"placeholder"` (not null; the schema gate uses the string to distinguish "needs filling in" from "actively malformed")
- Placeholder rows are committed at this phase for at minimum: (linux, 3.11), (linux, 3.12), (win32, 3.11) so Phase 42 CI runners can spot which combos still need backfill. Other combos (the maintainer's local (darwin, 3.11 or 3.12 depending on which is active) at minimum, plus optionally a populated (darwin, the other Python)) are populated at commit time by running the script.

Ordering: rows sorted by (os, python) tuple ascending so the JSON diff is deterministic across re-captures on different machines.

Idempotency: re-running `scripts/capture_v0_4_baseline.py` on the same (os, python) combo MUST replace the existing entry in place (delete-then-append-then-sort pattern from capture_v0_3_baseline.py:198-203). Re-running on a different (os, python) combo adds the new entry and preserves the old one. Placeholder rows remain placeholder rows until explicitly populated by a capture-script run on that target.
