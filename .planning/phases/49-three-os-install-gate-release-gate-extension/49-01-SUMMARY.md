## What landed

### Task 1: `horus-os plugins grant --all` flag

The v0.5 ergonomics improvement that the CI install-smoke-plugin job
needed. Instead of typing `grant <name> <cap>` once per capability, the
new `--all` flag reads the plugin's declared capability set from the
`plugin_capabilities` table (the user-approved set at install time)
and grants every one in a single call.

- `src/horus_os/__main__.py`: `pi_grant` argparse block uses
  `add_mutually_exclusive_group(required=True)` between the positional
  `capability` (`nargs='?'`, default=None — v0.4 shape preserved) and
  the new `--all` flag (`action='store_true', dest='grant_all'`).
- `src/horus_os/plugins/installer.py`: new
  `grant_all_capabilities(name, *, db) -> list[str]` helper reads
  `plugin_capabilities` rows for the installed version, calls
  `PermissionService.grant` once per capability (idempotent on
  re-runs per Phase 43 contract). Threat-register T-49-02 mitigation:
  reads from the persisted plugins table (user-approved set), NEVER
  from the on-disk manifest.
- `src/horus_os/cli/plugins_cmd.py`: `_cmd_grant` branches on
  `args.grant_all`. False -> v0.4 single-capability path byte-identical.
  True -> new path prints `Granted {n} capabilities to {name}: {caps}.`
  verbatim (the CI step greps for `Granted 2 capabilities`).
- `tests/plugins/test_plugins_grant_all.py`: 7 tests covering happy
  path (2 caps granted), idempotency, v0.4 positional regression
  (byte-identical message), argparse-level acceptance, mutex with
  positional, and missing-arg rejection.

Step list:
1. Check out repository
2. Set up Python (matrix)
3. Install horus-os editable + [dev]
4. Install reference plugin via `pip install -e ./examples/horus-os-example-plugin --no-deps`
5+6. Prepare isolated data dir + init horus-os DB (linux/macos + windows variants)
7+8. Launch horus-os serve in background (`nohup ... &` / `Start-Process -NoNewWindow`)
9+10. Poll `/api/system/ping` until ready (counter-based for-loop, 30 attempts at 1s)
11+12. Assert `/api/plugins` reports plugin as pending (Python one-liner asserts `status in ('pending','error')` + `set(pending_capabilities) == {'filesystem.read','secrets.read'}`)
13+14. Grant all capabilities via `python -m horus_os plugins grant horus-os-example-plugin --all --data-dir $HORUS_DATA` + grep for literal `Granted 2 capabilities`
15+16. Stop serve and relaunch (kill PID + nohup respawn + ping repoll)
17+18. Assert `/api/plugins` reports plugin as loaded (`status=='loaded'` + `granted_capabilities == both` + `pending_capabilities == []`)
19+20. Teardown (`if: always()`) — stop background serve + tail serve log on failure

## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 3 - Blocking issue] tests/test_release_gate.py::_write_ci_yml helper expanded**

- **Found during:** Task 2 (running pytest after adding the 4 new
  checks)
- **Issue:** Two existing v0.4 release-gate tests
  (`test_main_exit_zero_when_all_checks_pass` and
  `test_pricing_threshold_overridable_via_env`) used a shared
  `_write_ci_yml` helper that only emitted the two v0.4 job literals
  (`install-smoke-no-otel` and `install-smoke-with-otel`). After
  Task 2 the same tmp ci.yml is read by the new v0.5
  `plugin-install-smoke-ci` check, which then fails because the helper
  doesn't write `install-smoke-plugin`. The v0.4 tests exit with code
  1 (test failure) even though the v0.4 check function bodies are
  byte-identical.
- **Fix:** Added a `has_plugin_install: bool = True` kwarg to
  `_write_ci_yml`. The default value means the existing call sites
  silently start emitting the new job literal, restoring exit 0.
- **Files modified:** tests/test_release_gate.py
- **Commit:** 64ac6d3

**2. [Rule 1 - Test correctness] Replaced unrealistic data-loss test**

- **Found during:** Task 2 (running RED tests)
- **Issue:** The plan's `test_v0_4_fixture_roundtrip_fails_on_data_loss`
  test description called for a "SQLite without expected v6 state"
  that the check rejects. But `Database.init()` builds all v5/v6
  tables fresh via `CREATE TABLE IF NOT EXISTS` on any empty SQLite
  file — the synthetic empty-SQLite case always *passes* the check,
  not fails it. The plan's negative-path test was based on an incorrect
  model of the migration semantics.
- **Fix:** Replaced with
  `test_v0_4_fixture_roundtrip_fails_on_corrupt_sqlite` which writes
  the SQLite magic-header bytes followed by 4KB of `0xff` garbage.
  `sqlite3.connect` accepts the path lazily but the first PRAGMA query
  inside `Database.init()` raises `DatabaseError` — the realistic
  "fixture exists, opens, but is unusable" failure mode.
- **Files modified:** tests/test_release_gate_v0_5_checks.py
- **Commit:** 64ac6d3
