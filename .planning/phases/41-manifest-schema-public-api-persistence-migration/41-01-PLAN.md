NEW TABLE plugins (CREATE TABLE IF NOT EXISTS, lives in SCHEMA_SQL):
- name TEXT PRIMARY KEY
- version TEXT NOT NULL
- manifest_hash TEXT NOT NULL
- enabled INTEGER NOT NULL DEFAULT 1
- installed_at TEXT NOT NULL
- source TEXT NOT NULL CHECK (source IN ('entry_point', 'filesystem'))

NEW TABLE plugin_capabilities (CREATE TABLE IF NOT EXISTS, lives in SCHEMA_SQL):
- plugin_name TEXT NOT NULL
- plugin_version TEXT NOT NULL
- capability TEXT NOT NULL
- manifest_hash TEXT NOT NULL
- state TEXT NOT NULL CHECK (state IN ('granted', 'pending', 'revoked'))
- granted_at TEXT
- PRIMARY KEY (plugin_name, plugin_version, capability)
- FOREIGN KEY (plugin_name) REFERENCES plugins(name) ON DELETE CASCADE

NEW TABLE plugin_status (CREATE TABLE IF NOT EXISTS, lives in SCHEMA_SQL):
- plugin_name TEXT PRIMARY KEY
- status TEXT NOT NULL CHECK (status IN ('loaded','pending','error','disabled'))
- error_phase TEXT
- error_message TEXT
- last_seen TEXT
- FOREIGN KEY (plugin_name) REFERENCES plugins(name) ON DELETE CASCADE

NEW COLUMNS via ALTER TABLE ADD COLUMN inside the v5->v6 conditional block (BOTH wrapped in try/except sqlite3.OperationalError per the v4->v5 idempotency pattern at storage.py:201-212):
- ALTER TABLE llm_calls ADD COLUMN plugin_name TEXT      -- NULLABLE; NULL = horus-os core
- ALTER TABLE tool_invocations ADD COLUMN plugin_name TEXT   -- NULLABLE; NULL = horus-os core

NEW INDEX (CREATE INDEX IF NOT EXISTS, lives inside the v5->v6 conditional block AFTER the ALTER TABLE):
- idx_tool_invocations_plugin ON tool_invocations(plugin_name, created_at)

DO NOT add an `idx_llm_calls_plugin` index. OBSERVE-01 mandates only the tool_invocations index (rollup-query speed); llm_calls per-plugin aggregations are rare enough that the existing idx_llm_calls_created_at + filter is fine for v0.5. Adding a second index would be premature.

DO NOT add NOT NULL on the two new plugin_name columns. v0.4 rows have no value for these and SQLite refuses NOT NULL ADD COLUMN without a DEFAULT on a populated table. Both columns stay NULLABLE forever.

DO NOT add foreign keys referencing plugins(name) on llm_calls.plugin_name or tool_invocations.plugin_name. v0.4 rows have NULL there and the plugin may be uninstalled while its observability history remains. Soft reference only.

DO NOT DROP or RENAME any existing column or table. v0.4 readers depend on every existing column staying byte-identical (Pitfall 9).

<manifest_schema_canonical>
MANIFEST_V1_SCHEMA, MANIFEST_VERSION, and the validate_manifest contract, locked by .planning/research/ARCHITECTURE.md §"Pattern 1: Manifest model" and PITFALLS.md Pitfall 2.

MANIFEST_VERSION = 1 (module constant in src/horus_os/plugins/manifest.py). Any future bump is a deliberate, additive-only migration.

MANIFEST_V1_SCHEMA is a pydantic v2 BaseModel with these REQUIRED fields:
- manifest_version: int       -- MUST equal MANIFEST_VERSION (1); else raise with the exact message "manifest_version=N not supported by horus-os 0.5; please upgrade"
- name: str                   -- lowercase ASCII + hyphens, regex ^[a-z][a-z0-9-]*$; failure: "manifest line N: 'name' must be lowercase ASCII letters/digits/hyphens; got '<value>'"
- version: str                -- PEP 440 valid; parsed via packaging.version.Version; failure: "manifest line N: 'version' is not a valid PEP 440 version; got '<value>'"
- description: str            -- max length 200; failure: "manifest line N: 'description' must be ≤200 chars; got <len> chars"
- author: str                 -- non-empty
- license: str                -- non-empty; convention SPDX-style (not enforced as a closed enum; we accept any string but document SPDX in the reference plugin)
- horus_os_compat: str        -- MUST parse as packaging.specifiers.SpecifierSet via a pydantic field_validator; failure: "manifest line N: 'horus_os_compat' is not a valid PEP 440 specifier set (see packaging.specifiers.SpecifierSet); got '<value>'"
- contributions: ManifestContributions  -- nested model with two list[ContributionEntry] sub-fields (tools, adapters)
- capabilities: list[str]     -- each entry MUST be a value in capability_catalog.Capability; failure: "manifest line N: 'capabilities[i]' value '<value>' is not a member of the Capability catalog (allowed: filesystem.read, filesystem.write, net.outbound, secrets.read)"

OPTIONAL fields (default None or empty):
- homepage: HttpUrl | None    -- pydantic HttpUrl validator
- issue_tracker: HttpUrl | None

ManifestContributions submodel:
- tools: list[ContributionEntry]      (default [])
- adapters: list[ContributionEntry]   (default [])

ContributionEntry submodel:
- name: str                   -- regex ^[a-z][a-z0-9_]*$
- entry_point: str            -- dotted-path, regex ^[a-z][a-z0-9_]*(\.[a-z][a-z0-9_]*)*(:[A-Za-z_][A-Za-z0-9_]*)?$

Top-level unknown-fields handling per Pitfall 2:
- Set `model_config = ConfigDict(extra='ignore')` so unknown fields silently drop from the validated model.
- BUT also pre-scan the parsed TOML dict for unknown top-level keys (set difference against MANIFEST_V1_SCHEMA.model_fields) and emit a `UserWarning` for each: f"unknown manifest field {key!r}; this field is ignored under manifest_version={MANIFEST_VERSION}". DO NOT raise. Plugin authored against a future manifest_version still loads with warnings.

validate_manifest(toml_bytes: bytes) -> PluginSpec:
1. tomllib.loads(toml_bytes.decode('utf-8')) -> dict.
2. Pre-scan for unknown top-level keys; emit UserWarning per Pitfall 2.
3. MANIFEST_V1_SCHEMA.model_validate(dict_payload) -> validated model.
4. Translate validated model into a frozen PluginSpec (transform .model_dump() into the dataclass fields; convert capabilities str list into tuple[CapabilityRequest, ...] by looking up each name in Capability and constructing a CapabilityRequest with reason='' for now; richer per-capability reason fields land later if needed).
5. Compute manifest_hash via compute_manifest_hash(validated.capabilities); attach to PluginSpec.manifest_hash.
6. Return the PluginSpec.

format_validation_error(exc: ValidationError) -> str:
For each error in exc.errors(), produce one line of the form:
    "manifest field <dotted-loc>: <plain-English message>; got <repr(input)>"
Where:
- <dotted-loc> joins exc.errors()[i]["loc"] tuple with "." (e.g. "contributions.tools[0].name").
- <plain-English message> maps pydantic's type-code messages to humanized strings (a small dict literal in manifest.py: 'missing' -> "field is required", 'string_type' -> "must be a string", 'value_error' -> exc.errors()[i]["msg"] verbatim, ...). For unhandled type codes, fall back to exc.errors()[i]["msg"].
- <repr(input)> uses repr() on exc.errors()[i]["input"]; truncate to 80 chars + "..." if longer.
The format is plain-English and copy-paste-friendly; the user sees exactly what the loader saw and which field broke. Tested by test_manifest_schema.py asserting string substrings for each malformed fixture.

compute_manifest_hash(capabilities: Iterable[str]) -> str:
1. sorted_caps = sorted(set(map(str, capabilities)))
2. payload = "\n".join(sorted_caps).encode('utf-8')
3. return hashlib.sha256(payload).hexdigest()

The hash is capability-order-independent (sort), duplicate-tolerant (set), and changes when the set changes. Phase 43's plugin_capabilities.manifest_hash column persists this value; the PermissionGate compares the current manifest's hash against the persisted hash and flips state to 'pending' on mismatch.

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| plugin author → horus-os manifest parser | Plugin author writes `horus-plugin.toml`; horus-os parses it. Author-supplied data is structurally untrusted (e.g. malformed PEP 440 version, claim of unknown capability, oversize description). validate_manifest() is the enforcement point. |
| stored database → fresh process | A v0.4 database on disk is read by a v0.5 process; the database is trusted (the user's own data) but the migration must not corrupt it. v5->v6 ALTER TABLE statements are the enforcement point. |
| reference plugin / future third-party plugins → horus-os internals | A plugin imports from `horus_os`. Pitfall 8 demands the only legal import path is `horus_os.plugins.api`. The boundary is *defined* in this phase (the api.py module) but *enforced* in Phase 48 (ruff custom rule). |
