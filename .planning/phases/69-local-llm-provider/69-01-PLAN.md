<objective>
Add a local OpenAI-compatible LLM provider so a full agent turn runs with zero cloud API key. One adapter (`_openai_compat.py`) covers Ollama, llama.cpp server, vLLM, LM Studio, and OpenRouter via the `openai` SDK `base_url` override, mirroring the existing `_anthropic.py` / `_gemini.py` shape. Wire `provider="local"` into every provider-selection site (agent.py, server/api.py, run_cmd.py) and add the `local_base_url`, `local_model`, and `local_context_window` config fields.

Purpose: PROJECT.md core value #1 ("run on your machine, cloud never required to start") is unmet today because both shipped providers are cloud. This plan closes that gap and forces the LP-1 streaming/tool-call contract decision at provider-module creation time, before later phases depend on the provider layer.

Output: `_openai_compat.py` provider module; new `[local-llm]` extra; three config fields; provider-dispatch branches; TEST-32 mocked full-agent-turn regression test plus a provider unit-test module.
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| horus-os process to local model HTTP endpoint | The configured base_url is an outbound HTTP target the agent loop sends prompts to and trusts responses from. A non-loopback base_url silently widens this boundary onto the LAN. |
| config.toml / env to provider client | local_base_url and HORUS_OS_LOCAL_BASE_URL flow into the openai client constructor unvalidated in this plan; binding-address validation is enforced in plan 69-03 (doctor) per LP-4. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-69-01 | Information disclosure | local_base_url default points at the LAN | mitigate | Default base_url is the loopback `http://localhost:11434/v1`; the literal "0.0.0.0" never appears in defaults, config dump, or docs (LP-4). Active non-loopback rejection is delivered in plan 69-03 doctor. |
| T-69-02 | Tampering | Tool-call responses from an untrusted local server | accept | The local model is operator-configured and runs on the user's own machine; tool dispatch still flows through the existing ToolRegistry + allowed_tools filter, so no new privilege path is opened by this plan. |
| T-69-SC | Tampering | `openai>=2.40,<3` package install | mitigate | openai is an established, audited dependency (see RESEARCH Package Legitimacy Audit if present); it ships as an optional extra, not a base dependency. If RESEARCH.md lacks an audit row for openai, a blocking legitimacy checkpoint must precede the install. |
</threat_model>
