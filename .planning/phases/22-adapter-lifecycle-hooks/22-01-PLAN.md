# horus-os Phase 22 Plan 01

## Sibling Protocol shape

```
@runtime_checkable
class LifecycleAdapter(Protocol):
    name: str
    def start(self, context: AdapterContext) -> Awaitable[None]: ...
    def stop(self) -> Awaitable[None]: ...
```

The existing `Adapter` Protocol is unchanged. Runtime dispatch in
the lifespan handler uses `hasattr(adapter, "start")` and
`hasattr(adapter, "stop")` so an adapter only has to implement the
half it needs. The sibling Protocol exists for type hinting and
for the docstring story (third-party authors can declare
`class MyAdapter: ...; isinstance(MyAdapter(), LifecycleAdapter)`).

## AdapterRegistry

```
@dataclass
class AdapterEntry:
    name: str
    status: str = "stopped"  # running | stopped | error
    last_activity_at: str | None = None
    error_count: int = 0
    error_message: str | None = None


class AdapterRegistry:
    def __init__(self) -> None:
        self._entries: dict[str, AdapterEntry] = {}

    def register(self, name: str) -> AdapterEntry: ...
    def mark_running(self, name: str) -> None: ...
    def mark_stopped(self, name: str) -> None: ...
    def mark_error(self, name: str, message: str) -> None: ...
    def touch(self, name: str) -> None: ...
    def entries(self) -> list[AdapterEntry]: ...
```

`entries()` returns a list sorted by name. `touch` updates
`last_activity_at` to `datetime.now(timezone.utc).isoformat()`.
`mark_error` increments `error_count` and stores the message.

## AdapterContext addition

```
@dataclass(frozen=True)
class AdapterContext:
    config: Config
    data_dir: Path
    registry: AdapterRegistry = field(default_factory=AdapterRegistry)
```

`field(default_factory=AdapterRegistry)` keeps existing direct
constructions one-liner compatible. The dataclass is still frozen,
which is fine because `registry` is a frozen reference to a mutable
object.

## FastAPI lifespan integration

```
from contextlib import asynccontextmanager

def _make_lifespan(adapters, context):
    @asynccontextmanager
    async def lifespan(app):
        for adapter in adapters:
            start = getattr(adapter, "start", None)
            if start is None:
                continue
            try:
                await start(context)
            except Exception as exc:
                context.registry.mark_error(adapter.name, f"{type(exc).__name__}: {exc}")
        try:
            yield
        finally:
            for adapter in reversed(adapters):
                stop = getattr(adapter, "stop", None)
                if stop is None:
                    continue
                try:
                    await stop()
                except Exception as exc:
                    context.registry.mark_error(adapter.name, f"{type(exc).__name__}: {exc}")
    return lifespan
```

The discovery walk and registry population happen before the
`FastAPI(..., lifespan=...)` instantiation so the lifespan can close
over the resolved adapter list.

## create_app reshuffle

The current flow in `create_app`:

1. Build `FastAPI` instance
2. Register routes
3. Discover adapters and bind

Becomes:

1. Discover adapters via `discover_adapters()`
2. Build the registry, register every adapter with `status: stopped`
3. Build the `AdapterContext` carrying config, data_dir, registry
4. Build the `FastAPI` instance with `lifespan=_make_lifespan(adapters, context)`
5. Attach `app.state.adapter_registry = context.registry`
6. Register core routes (including `GET /api/adapters`)
7. Loop over adapters, call `bind(app, context)`; on success mark
   `running` (lifecycle hooks transition this further during
   lifespan); on bind exception, mark `error` and continue

Order rationale: discovery must come first because the lifespan
closes over the adapter list. The registry is populated at bind
time so `GET /api/adapters` returns useful data even before lifespan
fires.

## GET /api/adapters route

```
@app.get("/api/adapters")
def list_adapters(request: Request) -> dict[str, Any]:
    registry: AdapterRegistry = request.app.state.adapter_registry
    return {
        "adapters": [
            {
                "name": entry.name,
                "status": entry.status,
                "last_activity_at": entry.last_activity_at,
                "error_count": entry.error_count,
                "error_message": entry.error_message,
            }
            for entry in registry.entries()
        ]
    }
```

`registry.entries()` returns the sort-by-name list directly.

## Webhook activity touch

In `WebhookAdapter._handle_webhook`, after the successful
`db.record_trace(...)`, call `context.registry.touch(self.name)`.
This is the demonstration pattern for v0.3 adapters and proves
the registry path end-to-end.

## Status transitions on bind

- `bind` raises -> `mark_error(name, message)` and continue to next adapter
- `bind` returns -> `mark_running(name)`. Lifecycle hooks may
  override this further (e.g. if `start` raises, the entry flips
  to `error`)
- `WebhookAdapter` has no `start`, so after bind + lifespan, status
  remains `running`

## Public surface additions

`horus_os.adapters.__init__` re-exports `AdapterRegistry`,
`AdapterEntry`, and `LifecycleAdapter`. `horus_os.__init__` mirrors
those re-exports in `__all__` so `from horus_os import
AdapterRegistry, LifecycleAdapter` works.

## Test stubbing pattern

The lifecycle tests reuse the `_FakeEntryPoint` and stub helper
from `test_adapters_integration.py` (copied into the new file to
keep the tests self-contained). TestClient triggers the lifespan
on `__enter__` and `__exit__` so the end-to-end tests use:

```
with TestClient(create_app(data_dir=tmp_path)) as client:
    ...
```

The `with` block is what fires the lifespan on TestClient. Without
it, the lifespan does not run.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Adapter `start`/`stop` -> FastAPI lifespan | Third-party code runs at server startup with full Python privileges |
| Adapter handler -> AdapterRegistry | Adapters can write `last_activity_at` for any name they touch |
| `/api/adapters` route -> external callers | Lists adapter names and error messages |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-22-01 | Denial | Adapter `start` hangs and blocks the lifespan | accept | Same trust posture as Phase 17: adapters are pip-installed packages the operator chose. Documented in the adapter author guide |
| T-22-02 | Denial | Adapter `start` raises and blocks other adapters | mitigate | Lifespan wraps each adapter's `start` in try/except so siblings still get a turn |
| T-22-03 | Information Disclosure | `/api/adapters` leaks exception messages with paths or secrets | mitigate | error_message stores `type(exc).__name__: str(exc)`. Adapter authors are responsible for not embedding secrets in exception strings, same guidance as the rest of the API |
| T-22-04 | Tampering | Adapter overwrites another adapter's registry entry via `touch(other_name)` | accept | Adapters trust each other within the process; this is in-band cooperation, not a security boundary |
| T-22-05 | Denial | `stop` hangs on shutdown | accept | FastAPI lifespan exit cancellation handles a misbehaving stop; we do not add a wall-clock timeout in v0.3 |
</threat_model>
