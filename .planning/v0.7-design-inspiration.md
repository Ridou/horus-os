# v0.7 Design + Feature Inspiration

**Source:** read-only review of the private horus codebase at `/Users/santino/projects/horus` on 2026-05-30.
**Purpose:** Capture concrete patterns worth porting from private horus to open-source horus-os in milestone v0.7+.
**Scope:** Generic, open-sourceable patterns only. No PII, no Sentinel internals, no Vercel/Supabase coupling, no Google OAuth specifics (those land via existing opt-in adapters).

---

## TL;DR

The OSS horus-os dashboard feels plain because it is structurally absent compared to private horus:

| Surface | Private horus | OSS horus-os today |
|---|---|---|
| Dashboard pages | 75+ Next.js routes under `mission-control/app/dashboard/` | 6 tabs in a single-page HTML |
| Design system | Tailwind v4 + Radix UI + Recharts + Lucide, named dark theme with 10+ tokens | None. Inline CSS. No tokens. |
| Default agents | 8 named agents with personas, colors, domain | 1 generic default agent |
| Seed content | 2,200+ vault notes, sample tasks, demo traces | Empty on first run |
| Information architecture | Team + Memory + Tasks + Activity + Goals + ... | Flat tabs |

v0.7 should be a focused product-polish milestone: "Look and Feel + Starter Team". This doc captures the patterns to inspire that work.

Class-naming convention used in private horus: `<utility>-<group>-<role>` (e.g., `bg-bg-primary`, `text-text-secondary`, `border-border-subtle`, `bg-accent-cyan`). Doubled prefix is awkward but NativeWind-portable.

### 1b. Recommended stack for v0.7 frontend

- **Next.js 16, React 19** — same as private horus. Match the major versions so component patterns transfer.
- **Tailwind v4** — design tokens above.
- **Radix UI** — headless components for dialogs, menus, popovers, tooltips.
- **Recharts** — observability/cost charts (replace whatever the current HTML does).
- **Lucide React** — icon set. Private horus uses these icons consistently.

### 1c. Component patterns observed

From `mission-control/app/dashboard/team/page.tsx` (the team page):

- **Status filter tabs** with live counts: `[ All 8 | Active 3 | Running 1 | Paused 0 | Error 0 ]`
- **View toggle** for list vs org-chart visualization
- **Status dot** with pulse animation when running:
  ```tsx
  <span className="relative flex h-2.5 w-2.5">
    {running && (
      <span className="absolute inline-flex h-full w-full
                       animate-pulse rounded-full bg-blue-400 opacity-80" />
  ```

- **Page skeleton** loaders by variant: `<PageSkeleton variant="list" />`

### 1d. Page polish patterns from the memory page

From `mission-control/app/dashboard/memory/page.tsx`:

## Section 2: Information Architecture

### 2a. The "team" mental model

Private horus has a CEO + specialists organization that flows through the whole UI:

```
Coordinator/CEO routes work to:
├── Engineer (CTO domain)
├── Product (planning, roadmaps)
├── Researcher (deep dives)
├── Knowledge (memory, archiving)
├── Operator (runtime, monitoring)
└── Writer (docs, summaries)
```

Each agent has a color, a persona (`SOUL.md`), a rolling journal (`SESSION-LOG.md`), and a domain.

**OSS gap:** the multi-agent runtime exists (delegate_to_agent landed in Phase 13), but the dashboard's `/agents` page is a flat CRUD list with no team-as-organization mental model.

### 2b. The dashboard surface taxonomy

Mission Control has 75+ pages. After filtering out personal-data pages and integration-specific pages, the open-sourceable taxonomy looks like:

**Tier 1 — Core (port for v0.7):**
- `/team` — agent list + org chart toggle (replaces flat `/agents`)
- `/team/[agent]` — per-agent detail page (persona, sessions, color)
- `/memory` — markdown vault browser (search, folder tree, timeline)
- `/tasks` — task queue (pending, running, completed, retry/cancel)
- `/activity` — live timeline of agent actions
- `/traces` — already exists, redesign with new tokens
- `/costs` — already exists as `/observability`, redesign + chart upgrade
- `/settings` — first real settings page (currently nonexistent)
- `/about` — what is this, who runs it, how to ask for help

**Tier 2 — High-value, port if scope allows:**
- `/inbox` — items needing human approval (approve / reject / chat)
- `/approvals/[id]` — detail page for one approval request
- `/terminal` — live tail of an agent's stdout (watch it think)
- `/goals` — high-level objectives that agents work toward over time
- `/health` — system health (DB size, queue depth, error rate, uptime)
- `/schedules` — cron-like recurring agent runs
- `/onboarding` — first-run experience

**Tier 3 — Defer to v0.8+ or skip:**
- `/translator`, `/voice`, `/gemini` — requires specific paid SDK
- `/mail`, `/gcal`, `/calendar` — Google OAuth (already opt-in adapters)
- `/photos`, `/contacts`, `/reservations`, `/projects`, `/office` — personal data domains
- `/sentinel` — private pipeline; not for OSS
- `/yoga`, `/travel`, `/trending` — personal use cases

## Section 3: Starter Agent Team

### 3a. The SOUL.md persona pattern

Each private horus agent has a `SOUL.md` file. Test fixture example (`echo` / researcher) is 28 lines:

```markdown
---
name: echo
description: Research agent that gathers, analyzes, and summarizes
             external information for the team.
tools: []
allowed-tools: []
agent: researcher
plugin: core
updated: 2026-05-20
---

# Echo (test fixture) - SOUL

## Identity
You are **Echo**, the Research agent. You gather and analyze information
so {{USER_NAME}} can decide quickly.

## Principles
- Cite sources, even briefly.
- Prefer primary references over summaries.

## Voice
Crisp, citation-aware, no hedging.

## Boundaries
- Never fabricate citations.
- Never claim certainty beyond the evidence.

## Workflow
- Frame the question, search, synthesize, hand off.
```

**Pattern is portable:** YAML frontmatter for machine fields, 5 H2 sections (Identity, Principles, Voice, Boundaries, Workflow), `{{USER_NAME}}` template substitution. Generic enough to ship as a horus-os public pattern.

### 3b. Proposed OSS starter team

Generic agent names (not the private horus names which are personal). All ship with `horus-os init` auto-creation.

| Agent name | Domain | Color (matches private horus palette) | Default model |
|---|---|---|---|
| **Coordinator** | CEO/orchestrator, routes work | `#00d4ff` (accent cyan) | Sonnet 4.6 |
| **Engineer** | Code, technical implementation | `#22c55e` (success green) | Sonnet 4.6 |
| **Researcher** | Deep dives, analysis, citations | `#ec4899` (pink) | Haiku 4.5 |
| **Writer** | Docs, summaries, content | `#f59e0b` (warning amber) | Haiku 4.5 |
| **Operator** | Runtime, monitoring, alerts | `#a78bfa` (violet) | Haiku 4.5 |

Each gets a SOUL.md following the pattern above. Coordinator delegates to others via the existing `delegate_to_agent` tool from Phase 13.

### 3c. Seed content for first-run experience

- **Example vault:** `examples/example-vault/` with 10-15 generic markdown notes (meeting notes, project ideas, a journal entry, a TODO list, a "how to use horus-os" reference card).
- **Demo trace:** seeded on `horus-os init` so `/traces` is not empty. Banner: "(this is example data, run your own prompt to clear)".
- **Welcome dashboard:** first visit shows a `/onboarding` overlay tour pointing at each Tier-1 page in turn.

## Section 4: Memory Pipeline Inspiration

Private horus has 10+ specialized memory modules under `horus/`:

```
brain.py                 - high-level memory facade
daily_memory.py          - per-day summaries
memory_consolidation.py  - dedupe + merge facts
memory_engine.py         - LLM-driven fact extraction (1128 lines)
memory_recall.py         - retrieval-augmented context
place_memory.py          - geographic memory
solution_memory.py       - remember what worked
vault_drift.py           - detect changes in vault since last sync
vault.py                 - Obsidian vault read/write (843 lines)
working_memory.py        - short-term context
```

**OSS today** (Phases 5-6): a simple read/write markdown notes folder. No fact extraction, no consolidation, no entity-centric memory, no drift detection.

**Inspiration for v0.7+ (probably v0.8 scope):**

- **Fact extraction pipeline.** LLM reads new memory writes, extracts structured facts (entities, relations, dates), stores in SQLite. Surfaced in `/memory` as a "Knowledge" tab.
- **Daily consolidation.** Background job at end of day rolls up trace events into a daily-summary note. Visible in `/memory` Timeline.
- **Working vs long-term memory split.** Short-term context the coordinator passes to subagents, long-term the agent searches at retrieval time.
- **Vault drift detection.** When the user edits notes outside horus-os, the system notices and reindexes only what changed.

These are big features. Probably one whole milestone (v0.8 "Memory pipeline").

## Section 6: What NOT to Port

Explicit don't-port list so future scope discussions don't get confused:

1. **Vercel deploy + Supabase coupling.** Violates OSS "no required SaaS" principle. Static export is fine; hosted deploy is the user's choice.
2. **Sentinel pipeline.** Private. Per memory rule, never named or described in committed horus-os files.
3. **Personal agent names** (Ralph, Echo, Athena, Atlas, Quill, Scribe, Postman, Horus). Use generic names in OSS (Coordinator, Engineer, Researcher, Writer, Operator).
4. **Personal data domains** (yoga, reservations, contacts, projects, mail, gcal, calendar). The user's data, not the project's surface.
5. **Voice features via Gemini Live API.** Specific paid SDK. Could revisit as opt-in adapter in v0.8+.
6. **Discord 19-slash-command bot.** OSS Discord adapter is intentionally simpler (Phase 23 shipped a minimal version). Don't expand without good reason.
7. **JSON sync + git-deploy-on-push pattern.** Cool but Vercel-specific. OSS dashboard runs locally; static export is enough.
